Technology

How it actually works

Not just another screenshot → LLM → tap loop. OpenCyvis makes specific architectural choices that matter in practice.

01 — Perception

Dual-Channel Perception

Most phone agents rely solely on screenshots — the model "sees" the screen and guesses coordinates. This vision-only approach frequently misses targets on complex UIs.

OpenCyvis feeds both a pixel-level screenshot and the structured Accessibility Tree on every step. The screenshot provides semantic understanding; the UI tree provides precise coordinates.

Screenshot

Visual layout, colors, icons, text rendering. Answers: "What is this?"

UI Element Tree

Widget type, exact bounds, text content, hierarchy. Answers: "Where is it?"

Vision-only agents: ~60% tap accuracy on complex UIs
Dual-channel: precise coordinate targeting via a11y bounds
02 — Actions

Native Tool Calling

No regex parsing of free text. The model returns structured JSON via LLM function calling protocol — directly parseable, always verifiable.

14 action types: tap, swipe, type_text, open_app, ask_user, note, remember, handoff_user, wait, finish, fail, long_press, key_event
03 — Memory

Two-Tier Memory

Per-task notes carry information across steps (price found in app A, compared in app B). Cross-task remember persists user preferences in local DB.

note → working scratchpad, 10 slots, FIFO
remember → persistent memory, survives across tasks
04 — Display

VirtualDisplay Isolation

A real Android display created via system API — not a screen overlay or accessibility hack. Apps are migrated to it and run natively. Screenshot capture and input injection target this display independently.

DisplayManager.createVirtualDisplay()
IActivityTaskManager.moveTaskToDisplay()
InputManager.injectInputEvent(displayId)
05 — Philosophy

Minimal Engineering Constraints

No intent classifiers, no task templates, no workflow routers. The model itself understands intent, plans steps, and calls tools. The system scales with model improvements — no code changes needed.

A 1.8 GB local model (Gemma 4 E2B) passes all 4 test scenarios.
Hardcoded rules become tech debt as models outgrow them.

Design principle: Let the model do the thinking. Keep the engineering minimal. As foundation models improve, this architecture scales naturally — without rewriting task logic or adding orchestration layers.

Privacy & Security

Not a place for "trust us."

An AI agent with full phone access is one of the most privileged pieces of software you can run.

  • Screenshots and UI tree stay in memory — never written to disk
  • You choose the endpoint — self-hosted, private cloud, or fully local
  • No telemetry, no analytics, no phone-home — zero tracking code
  • Local model option — use Ollama and nothing leaves the device
  • Open source — security researchers, journalists, anyone can audit
  • 100% of code is public. Always.

Getting Started

Build from source

Requires AOSP 16 (API 36+) with platform key signing. See the AOSP deployment guide for device setup, symlinks, and signing.

# Clone and build
git clone https://github.com/opencyvis/opencyvis-phone.git
cd opencyvis-phone/android
./gradlew assembleRelease

# No device? Try the emulator
./scripts/deploy-emu.sh

# Configure — local Ollama (fully private)
adb shell am start -a android.intent.action.VIEW \
  -d "opencyvis://config?provider=ollama\
    &base_url=http://localhost:11434&model=gemma4:26b"
